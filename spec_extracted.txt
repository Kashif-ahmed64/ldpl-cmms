1. Executive Summary
Liberty Daharki Powers Ltd currently operates its plant maintenance operations using MP2, a legacy Computerized Maintenance Management System (CMMS) originally developed by Datastream, now discontinued. This software is outdated, runs on obsolete desktop infrastructure, lacks modern UI/UX, has no web or network access, and does not integrate with current operational workflows.
This document defines the complete specification for LDPL CMMS — a fully custom, modern replacement built specifically for Liberty Daharki Powers Ltd. It will be a desktop application running on the internal LAN, backed by a local PostgreSQL database, with role-based access, full audit trails, and all modules that MP2 provided plus significant modern enhancements.
Project Goals
Replace MP2 completely with a modern, maintainable, internally-owned system
Run entirely on Liberty internal LAN — no external internet dependency
Full role-based access control across all departments
Centralized PostgreSQL database on a dedicated on-site server
Modern desktop UI built with Electron + React (same stack as LDPL Operations Portal)
Complete audit trails for all actions, approvals, and changes
Offline-capable — works even when LAN has issues
Exportable reports in Excel and PDF formats

4.3 Module 3 — Preventive Maintenance (PM) Scheduling
Preventive Maintenance is the proactive scheduling of maintenance tasks before equipment fails. This module auto-generates Work Orders when PM is due.
4.3.1 PM Schedule Types
Time-based — every X days/weeks/months (e.g. monthly oil change)
Meter-based — every X hours/KM/cycles (e.g. every 500 hours running time)
Condition-based — when a parameter exceeds a threshold
Seasonal — annual or bi-annual overhaul

4.3.3 PM Automation
System checks daily at midnight for PMs due within the lead time window
Auto-generates PM Work Orders and assigns to the configured department

4.10 Module 10 — Audit Trail & Security
Every action in the system is logged with timestamp, user, and the before/after values. This module is critical for compliance and accountability.
4.10.1 Security Measures
Passwords hashed with bcrypt (minimum cost factor 12) — never stored in plain text
JWT tokens with 8-hour expiry — auto logout on expiry
Refresh token rotation — new token issued on each request
Role-based middleware on every API endpoint — server-side enforcement
SQL injection prevention via Prisma ORM parameterized queries
Input validation and sanitization on all form fields
Rate limiting on login endpoint — 5 failed attempts locks account for 15 minutes
HTTPS on LAN using self-signed certificate
Database connection encrypted
Nightly encrypted database backup to external HDD
